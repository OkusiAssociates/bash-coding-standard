## 1. Headline

The corpus is **mechanically pristine** but **content-skeletal**. 67 % of leaves
need work before the document can serve as a reference; 30 % need substantive
prose authoring (PROMOTE); 41 % need targeted enrichment (ENRICH). KEEP-rate
of 28 % is healthier than expected — driven entirely by the P3 specialist
Parts (XVIII Readline, XXIII POSIX, XXIV Internals, XXV 5.3-future) and the
Appendices, where bullet-form is the correct form.

350 leaves contain any fenced code block (8 %), concentrated in Part XXII
(50 %). Eighteen Parts contain **zero** code in any leaf — including Parts
III–IX, the language-proper section that most needs worked examples to
function as a reference.

The single biggest cross-cutting strength is **structural integrity**:
zero broken cross-references across 124 `§N.M(.K)` citations, zero heading
mismatches, zero missing `#fin` markers, exact filename–TOC alignment.

## 6. Twenty-five highest-leverage findings

Ranked by reader-impact × frequency × P1-status. Every finding is
sourced from a per-Part audit; chase those for surrounding detail.

### Critical (correctness / safety)

1. **§13.2 `set -e` semantics**, **§13.3 errexit-exemption matrix** — single
   most-cited error-handling reference; current skeleton cannot resolve
   the row-by-row exemption rules readers come for. PROMOTE 200+ lines
   each.
2. **§16.5 buggy idiom** — `pids=("${pids[@]/$done_pid}")` uses
   substring-replacement, not array-element removal. `[critical]`
   factual error in source. Auto-fix during ENRICH pass.
3. **§02.8 EXIT-trap inheritance claim** — current text confuses `set -E`
   (ERR-trap) with EXIT-trap inheritance. `[major]` factual error.
4. **§07.1 form-count mismatch** — claims "one of seven forms" then
   lists ten. `[fixable]`.
5. **§13.9 strict-mode contract drift** — leaf says `set -eET -o pipefail`
   + `inherit_errexit`; BCS canonical is `set -euo pipefail` with
   `shopt -s inherit_errexit shift_verbose extglob nullglob`. Reconcile.

### High-leverage PROMOTE candidates

6. **§20.5 Command-injection vectors** — central P1 chapter; bullets
   without vulnerable/fixed pairs are dangerous.
7. **§22.3 Argument-parsing skeleton** — for an *idioms* cookbook,
   deferring the canonical parser is breaking the contract.
8. **§22.15 Stack-trace error reporter** — canonical FUNCNAME/BASH_LINENO
   walker missing despite five back-references.
9. **§14.7 Logging discipline** — BCS `_msg`+FUNCNAME pattern is
   canonical and absent from a leaf that's a P1 anchor.
10. **§04.5 `declare` builtin and attributes** — 19 lines for the
    BCS-central type system.
11. **§04.11 Namerefs (`-n`)** — output-parameter pattern and
    `local -n self=self` pitfall described in prose only.
12. **§05.8 Word-splitting and `IFS`** — 14 lines for the #1 source of
    production bash bugs.
13. **§06.8 Heredocs** — quoted/unquoted/`<<-` rules unverifiable from
    prose alone.
14. **§06.11 Redirection order-of-evaluation** — the canonical
    `>file 2>&1` vs `2>&1 >file` trace has no demonstration.
15. **§15.11 Auto-generating usage** — deferred-action / SSoT pattern
    central to the BCS template; gestured-at, not implemented.
16. **§20.13 Symlink races** — must show canonical `mktemp -d` +
    trap-cleanup wrapper.
17. **§20.4 `eval` avoidance** — needs the nameref / `declare -A`
    refactor of `eval "var_$key=$value"` shown explicitly.
18. **§16.10 Locking primitives** — canonical
    `(flock -x 200; ...) 200>FILE` subshell pattern is missing entirely.
19. **§21.07 CI integration** — actual GitHub Actions + GitLab YAML.
20. **§21.10 Bats run + assertions** — `run` + `assert_*` worked example.

### Cross-cutting structural

21. **Code-block desert** — 18 Parts with zero code blocks. For a bash
    *reference* this is the highest-impact structural defect.
22. **Reciprocal cross-refs missing** between adjacent semantic
    neighbours: §16.10 ↔ §20.10 (lockfiles); §16.9 ↔ §20.13 (races);
    §10.2 ↔ §9.11 (`BASH_SOURCE`); §18.13 ↔ §19.5 (PS4); §19.4 ↔ §13/14
    (xtrace fd vs error-handling).
23. **BCS rule codes never inlined** — RAG retrieval will not surface
    relevant standard rules from the corpus. PROMOTE expansions should
    cite codes inline (`BCS0203`, etc.) per BCS-bash convention.
24. **Strict-mode framing absent** in Parts XI / XVI / XIX / XXIV —
    the engineering layer where strict-mode interaction is most subtle.
    Recommend a mandatory "Strict-mode interaction" stanza in every
    PROMOTE leaf in those Parts.
25. **Self-containment risk for AI/RAG** — body chapters cite
    "see BCS §4.5" and "see Greg's wiki on word splitting" without
    inlining; a RAG retriever lacks the resolver. Decision: see
    `self-containment-issues.md`.
