### BCS1002 — PATH Security  `[consistency]` (10-security.md)

**Problem:** The prose says "Never include `.`, empty elements (`::`, leading/trailing `:`), `/tmp`, or user home directories in PATH", but the first "correct" example is `declare -rx PATH=~/.local/bin:/usr/local/bin:/usr/bin:/bin` — which puts a user home directory (and first in search order, no less) into PATH. An LLM checker reading this rule cannot decide whether `~/.local/bin` in PATH is a violation or the blessed pattern.

**Suggestion:** Pick one position. Either remove `~/.local/bin` from the correct example, or qualify the prose: "Never include `.`, empty elements, or `/tmp`. **Verifier note:** Verified at /ai/scripts/Okusi/BCS/data/10-security.md: line 33 marks `declare -rx PATH=~/.local/bin:/usr/local/bin:/usr/bin:/bin` as '# correct' (home-dir path first in search order), while line 43 states 'Never include `.`, empty elements (`::`, leading/trailing `:`), `/tmp`, or user home directories in PATH.' The suggested fix (qualify the prose to permit user-writable dirs only in non-privileged tooling, forbidding them in production/sudo contexts) matches the distinction the examples already draw, preserves both examples, and is K.I.S.S.-compatible. Severity high is appropriate: direct self-contradiction inside a core security rule affecting a high-frequency pattern.

### BCS1005 — Input Sanitization  `[example-quality]` (10-security.md)

### BCS1102 — Parallel Execution  `[consistency]` (11-concurrency.md)

**Problem:** The 'correct — parallel with ordered output' example contains `wait "${pids[0]}" ||:`, which discards the exit code of `wait`. This directly violates BCS1103 (core) in the same file: 'Never discard the exit code of `wait`.' Every server check could fail and the script would still print output and exit 0. The standard's own canonical example commits a core violation, which will confuse both human readers and the LLM checker (a script copying this pattern would be flagged under BCS1103).

**Suggestion:** Make the example BCS1103-compliant: declare `declare -i errors=0` before the loop, replace `wait "${pids[0]}" ||:` with `wait "${pids[0]}" || errors+=1`, and end with `((errors == 0)) || die 1 "$errors job(s) failed"`.

**Verifier note:** Verified in /ai/scripts/Okusi/BCS/data/11-concurrency.md: BCS1102's '# correct — parallel with ordered output' example (line 65) contains `wait "${pids[0]}" ||:`, which discards the wait exit status — exactly what BCS1103 (core, line 79: 'Never discard the exit code of wait') forbids, and semantically identical to BCS1103's own '# wrong — exit code discarded; failures silent' example. The behavioral claim is correct: all server checks could fail and the script would still cat outputs and exit 0. Because the standard is fed verbatim to the LLM checker, a script copying this canonical example would be flagged as a core [ERROR] under BCS1103 — a real internal contradiction, not taste. The suggested fix mirrors BCS1103's prescribed accumulator pattern (declare -i errors=0; wait ... || errors+=1; ((errors == 0)) || die 1 ...) and fits BCS conventions (die, errors+=1, no ((count++))). No refutation holds; a maintainer should apply the fix.

### BCS1209 — Testing Support  `[example-quality]` (12-style-development.md)

**Problem:** The test-runner example is broken Bash: `while IFS= read -r _ _ fn; do` — with `IFS=` (empty) no field splitting occurs, so the entire `declare -f name` line lands in the first `_` and `fn` is always empty (verified: fn=[] for every input line). `if "$fn"` then executes an empty command name, so every test is counted as failed. Additionally `grep 'test_'` matches any function whose name merely *contains* `test_` (e.g. `latest_results`).

**Suggestion:** Change the loop to `while read -r _ _ fn; do` (default IFS so the three fields split) and tighten the filter to `done < <(declare -F | grep ' test_')` or `grep -E '^declare -f test_'`.

**Verifier note:** Verified empirically in bash: with `IFS= read -r _ _ fn`, empty IFS disables field splitting, so the whole `declare -f name` line lands in the first `_` and `fn` is empty on every iteration (observed fn=[] for all lines). `if "$fn"` then executes an empty command name (exit 127), so the example test runner counts every test as failed — the `# correct` example at data/12-style-development.md:216-229 is non-functional. Also confirmed `grep 'test_'` falsely matches `latest_results` (substring in `latest_`); `grep ' test_'` correctly matches only test functions. The suggested fix (default-IFS read + anchored grep) was tested and works, keeps the process-substitution form required by BCS, and removes nothing. Since bcs check feeds this document to LLMs as ground truth, a broken canonical example is a real defect; severity high stands.
