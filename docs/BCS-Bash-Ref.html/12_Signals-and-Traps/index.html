Part XII — Signals and Traps

Signals are bash's primary mechanism for asynchronous communication and lifecycle hooks. This Part documents the signal catalogue, the trap builtin, the pseudo-signals, and the discipline required to write signal-safe code.

Chapters

  1. 12.1 Signal taxonomy — Signals fall into broad functional categories.
  2. 12.2 Signal numbers and names — The mapping is platform-specific but stable on Linux.
  3. 12.3 Uncatchable signals — Two signals cannot be caught, blocked, or ignored.
  4. 12.4 Signal disposition — Each signal has one of four dispositions per process.
  5. 12.5 The trap builtin — Registers handler commands for signals and pseudo-signals.
  6. 12.6 Pseudo-signals: EXIT, ERR, DEBUG, RETURN — Bash extends signals with four pseudo-signals tied to script lifecycle events.
  7. 12.7 trap -p and trap inspection — Listing the current trap state.
  8. 12.8 Trap inheritance — Subshells reset most traps; functions inherit unless set -E/set -T.
  9. 12.9 Trap reset across exec — On exec, signal handlers are reset (POSIX requirement).
  10. 12.10 Synchronous vs asynchronous delivery — Bash delivers signals between commands, not mid-command.
  11. 12.11 Signal-safe code — Within a trap handler, certain operations are unsafe.
  12. 12.12 Idempotent cleanup patterns — Traps that must run at most once and produce the same effect on every invocation.
  13. 12.13 Tempfile and tempdir lifecycle — The canonical pattern for safe temporary storage.
  14. 12.14 Lockfile pattern — Mutual exclusion across script invocations.
  15. 12.15 Atomic file write — Write to a sibling tempfile, then rename.
  16. 12.16 Reload-on-SIGHUP — Convention: SIGHUP requests "reload your config".
