Part XXIII — POSIX Conformance and Portability

When and how to write code that runs on more than just Bash. Most production code does not need to be portable; some does. This Part documents the trade-offs.

Chapters

  1. 23.1 Bash vs POSIX sh — The features bash adds beyond POSIX 1003.2 / SUSv4.
  2. 23.2 The bashisms list — Specific constructs that fail in dash / POSIX sh.
  3. 23.3 Bash vs dash — dash is Ubuntu/Debian's /bin/sh — POSIX-only, no bashisms.
  4. 23.4 Bash vs ksh — Korn shell variants.
  5. 23.5 Bash vs zsh — zsh is interactive-rich, scripting-divergent.
  6. 23.6 Bash 3.2 on macOS — Apple ships bash 3.2 (2007).
  7. 23.7 BSD sh — FreeBSD, OpenBSD, NetBSD use various sh implementations.
  8. 23.8 --posix mode — Bash's POSIX-conformance mode.
  9. 23.9 shopt compatibility levels — Bash supports limited backward compatibility via shopt -s compatNN.
  10. 23.10 When to write portable sh — Cases where POSIX-only is the right choice.
  11. 23.11 Forward-compatibility hygiene — Writing bash that won't break in future versions.
  12. 23.12 Targeting multiple Bash versions — Supporting both old and new bash from one script.
